# Стилистика кода в MoonWalk
Здесь описаны правила для написания единобразного кода в рамках проекта MoonWalk.

## Комментарии
В этом разделе описаны правила по написанию комментариев и документации кода

### Уведомление о лицензии
Внутри каждого файла в самом начале (1 строка) указывается этот хеадер
```rust
// Часть проекта MoonWalk с открытым исходным кодом.
// Лицензия EPL 2.0, подробнее в файле LICENSE. UpdateDeveloper, 2025
```

### Над функциями, структурами и перечислениями
Используем `///` (Три слэша) для описания сущности и значений внутри

**Пример:**
```rust
/// Эта функция нужна для вывода тестового сообщения в терминал.
/// Она принимает аргумент test_arg (булевое значение) которое 
/// используется для уточнения при выводе.
fn test(test_arg: bool) {
    println!("Какой-то текст, аргумент: {}", test_arg);
}
```

> Уточнение: Если сущность использует значения с типом из какого-либо крейта необходимо указать из какого.

**Мы используем уточнения**
- [*] Информация
- [?] Ответ на вопрос
- [!] Предупреждение

### Комментарии внутри сущностей
Запрещено использовать многострочные комментарии, комментарии пишутся по этому шаблону

```rust
/// Документация
pub enum TestEnum {
    One = 1, // Описание
    
    // Дополнительная группа
    Two = 2,
}
```

## Подключение модулей
Всё использование `mod` и `use` делается СТРОГО в начале файла до объвления каких-либо структур/функций/перечислений/констант. Подключение и использование НЕ КОММЕНТИРУЕМ!

**Используем шаблон:**
```rust
// Сначала публичные модули, потом приватные (Без переноса строки)
pub mod a;
mod b;

// Тут также сначала публичные, потом приватные. Перенос строки после
// подключения модулей. Сначала просто использование, потом use crate::*
// потом уже use super::* С ПЕРЕНОСОМ СТРОК!
use a::c;

use crate::d::f;
```

## Стилистика тела ( {} )
В структурах, перечислениях и функциях используем стилистику K&R, то есть
```rust
fn lalala() { // ПЕРЕНОС СТРОКИ ПОСЛЕ '{'

}
```

## Рекомендации:
- Писать функции в теле которых не более 40 строк кода
- Не использовать .unwrap(), а использовать match либо .expect("...")
- Не сокращать несколько действий в 1 строку
